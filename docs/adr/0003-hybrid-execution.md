# ADR-0003: Hybrid execution — real VM + simulated pods

## Status

Accepted

## Context

A cyber range needs both realism and scale. Real virtual machines give players
hands-on experience with actual operating systems, services, and vulnerabilities.
But running a VM for every service in a city of 300+ services is impractical in
a home lab and wasteful even in production.

The engine must support a hybrid model where some services are real workloads
and others are simulated by the engine.

## Decision

Each service has a **runtime mode**:

- `real` — external VM or container; answers events through an agent.
- `simulated` — lightweight response generated by the engine emulator.
- `decoy` — a simulated honeypot with a deliberate fingerprint.

Mode is a deployment-time concern, not part of the canonical city data model.
The topology graph describes *what* exists; deployment configuration decides
*how* each service runs.

## Discovery and health

Real services run a small `cybercity-agent` that:

1. Reads `service_id` and connection settings from metadata (cloud-init, env).
2. Sends periodic `heartbeat` events to the event bus.
3. Listens for events addressed to `service_id`.
4. Reports scan/attack results back to the engine.

If a real service stops sending heartbeats, the engine marks it `down`.

## Simulated services

The engine's emulator generates plausible responses based on:

- service `kind` (web, db, dns, etc.);
- `ports` and `software` from topology;
- `decoy` fingerprint if present;
- current runtime state (up, down, compromised).

Simulated services are cheap and scale horizontally in Kubernetes.

## Decoys

Decoys are always simulated. They exist to attract scans and attacks, collect
threat intelligence, and slow down attackers. The emulator can generate
different personas: default creds, known CVEs, realistic banners.

## Consequences

### Positive

- Realism where it matters, scale where it does not.
- Home lab feasible with a handful of VMs.
- Training scenarios can mix real exploitation with simulated city effects.
- Easy to upgrade a service from simulated to real without changing topology.

### Negative

- Two execution paths to maintain.
- Real service agents need packaging and distribution.
- Network must reach both VMs and pods uniformly.

## Mapping example

```yaml
# deployment/home-lab/service-mapping.yaml
runtime_mode:
  bank-web: real
  bank-db: real
  bank-log: simulated
  hospital-pacs: real
  power-scada: real
  decoy-printer-01: decoy
  # everything else defaults to simulated
```

## Alternatives considered

- **All real**: too expensive, too much maintenance.
- **All simulated**: lacks hands-on value for players.
- **Real per organization**: simpler but still wasteful for tiny orgs.
