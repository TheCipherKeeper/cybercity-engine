# AGENTS.md — правила для AI-агентов и контрибьюторов CyberCity

## Иерархия документов (от старшего к младшему)

1. `master.md` — философия и дорожная карта. Не редактируется без пересмотра принципов.
2. `docs/adr/` — действующие архитектурные решения. ADR со статусом `superseded` не имеют силы.
3. `AGENTS.md` (этот файл) — операционные правила работы в репозитории.
4. `network.md` — реестр организаций и decoy-хостов (человекочитаемая проекция `network.yml`).
5. Всё остальное (код, схемы, генераторы) — реализация решений, принятых выше.

Если документы противоречат друг другу, побеждает старший. Любое расхождение — повод создать новый ADR, а не «тихо» править младший документ.

## Ключевые принципы (выжимка из master.md)

- **События — единственный источник истины.** Состояние = проекция потока событий.
- **Сеть декларативна.** Один файл (`network.yml`) описывает организации, сервисы, decoy-хосты и каналы. K8s-манифесты — его проекция (ADR-0007).
- **Безопасность по умолчанию.** Сегменты изолированы, каналы объявляются явно. OT никогда не светится наружу.
- **LLM — помощник, не хозяин.** LLM пишет YAML, код валидирует, человек решает.
- **Итеративная генерация.** По одной организации за раз, с растущим контекстом и обязательной валидацией.
- **Воспроизводимость.** Один вход → один выход. Детерминированный режим, фиксированный порядок ключей.

## Правила для AI-агентов

### Что агенту МОЖНО

- Писать Go-код в `cmd/`, `internal/`, в `scripts/` (скрипты — на Python или Go).
- Редактировать `network.yml` и другие YAML-файлы, **если задача явно про это**.
- Предлагать изменения в `network.md` (это реестр-проекция, не источник истины).
- Создавать новые ADR, если меняется архитектурное решение.
- Запускать тесты, линтеры, валидаторы.

### Чего агенту НЕЛЬЗЯ

- Редактировать `master.md` без явного указания пользователя. Этот файл описывает философию — его пересмотр = пересмотр принципов.
- Удалять или «чистить» ADR со статусом `superseded`. Они остаются в истории.
- Добавлять зависимости, не разрешённые ADR'ами. Каждая новая зависимость = новая запись в ADR или ссылка на существующую.
- Менять module path в `go.mod` без согласования.
- Делать коммиты, пуши, PR — это делает человек.
- Писать «защитный» код в обход валидатора. Если кажется, что валидатор слишком строгий — это повод обсудить правило, а не обходить его.
- Молча «творить» сразу 30 организаций. Любая итеративная генерация идёт по одной сущности за шаг.

### Рабочий цикл (когда задача касается network.yml)

См. раздел «Рабочий цикл» ниже — единый цикл для organizations и decoys.

## Структура репозитория

```
cybercity/
├── master.md           # философия и дорожная карта
├── network.md          # реестр организаций + decoys (проекция network.yml)
├── AGENTS.md           # этот файл
├── go.mod              # Go module
├── cmd/                # точки входа (validate-network, render-manifests, симулятор, CLI)
├── internal/           # реализация
│   ├── events/         # событийное ядро (ADR-0004)
│   ├── network/        # модель network.yml + единый валидатор (организации + decoys; ADR-0007)
│   ├── sim/            # движок симуляции
│   ├── runtime/        # K8s-адаптеры
│   └── scenario/       # пакеты сценариев
├── network.yml         # единственный канонический YAML (организации + decoys; ADR-0007)
├── network/            # сгенерированные K8s-манифесты
│   └── generated/      # в .gitignore
├── provisioning/
│   └── ansible/        # плейбуки для Proxmox-VM (ADR-0001)
├── scripts/            # вспомогательные скрипты
└── docs/
    └── adr/            # архитектурные decision records
```

## Рабочий цикл (когда задача касается network.yml)

1. Прочитать текущий `network.yml` (когда появится) и сжатый индекс уже
   описанных организаций + decoy-хостов.
2. Сгенерировать YAML для **одной** записи — организации или decoy.
3. Прогнать валидатор: `go run ./cmd/validate-network`.
4. Если ошибка — исправить, прогнать ещё раз. Не накапливать несколько
   изменений перед проверкой.
5. Показать результат пользователю. Не коммитить.

Единый валидатор покрывает все три фазы (ADR-0007 п. 2): структурную,
внутри-organization и кросс-фазу между organizations и decoys. Отдельного
`validate-decoys` нет.
